## Процесс работы

Основная ветка разработки: `develop`.
Дополнительные поддерживаемые ветки релизного цикла: `testing`, `main`.

### 1. Подготовьте форк и upstream

```bash
# Клонировать свой форк
git clone https://github.com/YOUR_USERNAME/poker-planning.git

# Перейти в директорию проекта
cd poker-planning

# Добавить оригинальный репозиторий как upstream
git remote add upstream https://github.com/your-org/poker-planning.git
```

### 2. Создайте отдельную ветку

```bash
# Получить актуальные изменения из upstream
git fetch upstream

# Создать ветку от upstream/develop
git checkout -b feature/short-description upstream/develop
```

Допустимые префиксы веток:

<div align="center">

| **Префикс**  | **Назначение**                      |
| :----------- | :---------------------------------- |
| `feature/*`  | Новая функциональность              |
| `fix/*`      | Исправление ошибок                  |
| `docs/*`     | Изменения в документации            |
| `refactor/*` | Рефакторинг без изменения поведения |
| `test/*`     | Тесты и тестовая инфраструктура     |
| `chore/*`    | Служебные задачи                    |

</div>

### 3. Держите ветку актуальной

```bash
# Обновить ссылки на ветки upstream
git fetch upstream

# Перебазировать текущую ветку на upstream/develop
git rebase upstream/develop
```

## Требования к изменениям

- Один Pull Request должен решать одну задачу.
- Изменения должны быть локальными и понятными по объему.
- Новая функциональность должна сопровождаться тестами.
- Если поведение изменено, документация должна быть обновлена.
- Нельзя добавлять секреты, токены, пароли и приватные ключи.

### Код-стиль (кратко)

<div align="center">

| **Стек**    | **Ожидание**                                                              |
| :---------- | :------------------------------------------------------------------------ |
| Backend     | Контракты API/WebSocket должны быть предсказуемыми и обратно совместимыми |
| TypeScript  | Строгая типизация, без неявных `any`, предсказуемые пропсы и контракты    |
| React       | Компоненты с понятными пропсами, минимизация re-render                    |
| Комментарии | Объясняют причину решения, а не пересказывают код                         |

</div>

## Тестирование изменений

Перед PR убедитесь, что:

- тесты проходят локально;
- линтеры и проверки стиля проходят;
- изменение покрыто тестами на уровне, достаточном для регрессий.

### Минимальный набор команд перед PR

```bash
# Из корня репозитория
pnpm format:check
pnpm lint

# Точечные проверки (по инструментам)
pnpm lint:eslint
pnpm lint:style

# Опционально: автоисправление
pnpm lint:fix
pnpm format

# Опционально: точечные автоисправления frontend
pnpm --filter @poker/frontend lint:eslint:fix
pnpm --filter @poker/frontend lint:style:fix
```

Если есть ошибки форматирования или стиля, исправьте их до создания PR.

## Pull Requests

### Чек-лист перед открытием PR

- [ ] Ветка синхронизирована с upstream/develop
- [ ] Лишние/временные файлы не добавлены
- [ ] Добавлены или обновлены тесты
- [ ] Обновлена документация при изменении поведения
- [ ] Изменения проходят локальные проверки
- [ ] `pnpm format:check` и `pnpm lint` проходят без ошибок
- [ ] При необходимости выполнен `pnpm lint:fix`

### Шаблон описания PR

```markdown
## Описание

Что было сделано и почему

## Type of change

- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Как проверялось

Кратко опишите сценарии проверки

## Чек-лист

- [ ] Код соответствует стилю проекта
- [ ] Добавлены нужные тесты
- [ ] Обновлена документация
- [ ] Все проверки проходят

## Related issue

Closes #123
```

### После открытия PR

1. Автопроверки CI должны быть зелёными.
2. Требуется минимум одно ревью и одобрение.
3. Замечания ревью должны быть закрыты до merge.

### Что проверяет CI

- Frontend: `pnpm lint`, `pnpm typecheck` и `pnpm build`
- Backend: smoke-проверки контрактов REST/WebSocket и инфраструктурные проверки

## Commit сообщения

Используйте формат Conventional Commits:

```text
<type>(<scope>): <subject>
```

Рекомендуемые type:

<div align="center">

| **Type**   | **Назначение**                      |
| :--------- | :---------------------------------- |
| `feat`     | Новая функциональность              |
| `fix`      | Исправление ошибки                  |
| `docs`     | Изменения в документации            |
| `refactor` | Рефакторинг без изменения поведения |
| `test`     | Добавление или обновление тестов    |
| `chore`    | Служебные изменения и обслуживание  |
| `perf`     | Оптимизация производительности      |
| `style`    | Форматирование и stylistic-правки   |

</div>

Рекомендуемые scope:

<div align="center">

| **Scope**    | **Примеры**                      |
| :----------- | :------------------------------- |
| `app`        | Роуты, инициализация, провайдеры |
| `pages`      | Страницы (Home, Room, Results)   |
| `components` | UI-компоненты                    |
| `features`   | Сценарии (VoteCard, CreateRoom)  |
| `entities`   | Сущности (Room, User, Card)      |
| `shared`     | Общие модули (api, ui, lib)      |
| `ws`         | WebSocket события                |

</div>
